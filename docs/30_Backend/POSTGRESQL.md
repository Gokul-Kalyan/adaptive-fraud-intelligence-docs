# 1. Purpose

Machine learning models generate predictions, but production systems must also persist those predictions for monitoring, auditing, analytics, and future model improvement.

The Adaptive Fraud Intelligence Platform integrates PostgreSQL as its persistent storage layer. Every prediction generated by the FastAPI service is stored in a relational database, enabling downstream components such as the monitoring dashboard and drift detection module to access production inference data.

This document describes the PostgreSQL architecture, database design, connection management, prediction logging workflow, and integration with the overall platform.

---

# 2. Why PostgreSQL?

Several storage options were considered for recording prediction history, including CSV files, SQLite, and PostgreSQL.

PostgreSQL was selected because it provides:

- Reliable relational data storage.
- ACID-compliant transactions.
- Efficient querying capabilities.
- Scalability for production workloads.
- Native integration with Python.
- Compatibility with Docker deployment.

Unlike temporary file-based storage, PostgreSQL provides persistent storage suitable for production environments.

---

# 3. Database Architecture

PostgreSQL serves as the central persistence layer of the platform.

```
                FastAPI
                   │
                   ▼
          Fraud Prediction
                   │
                   ▼
           Decision Engine
                   │
                   ▼
          Prediction Logger
                   │
                   ▼
         PostgreSQL Database
          (fraud_platform)
          ┌────────┴────────┐
          ▼                 ▼
 Streamlit Dashboard   Drift Detection
```

Instead of storing predictions in memory or CSV files, every inference result is written to the database.

The database therefore acts as the communication layer between multiple backend components.

---

# 4. Database Configuration

The application establishes database connections using the `psycopg` PostgreSQL driver.

Connection parameters are maintained separately through configuration variables.

```python
connection = psycopg.connect(
    dbname=DB_NAME,
    user=DB_USER,
    password=DB_PASSWORD,
    host=DB_HOST,
    port=DB_PORT
)
```

Separating configuration from application logic improves maintainability and simplifies deployment across different environments.

---

# 5. Database Initialization

During the initial project setup, the PostgreSQL database and the `predictions` table were manually created.

The application assumes that the required database schema already exists before the API starts.

Once initialized, the application performs only:

- INSERT operations
- SELECT operations

No automatic schema creation or migration is currently implemented.

---

# 6. Database Schema

Database Name

```
fraud_platform
```

Primary Table

```
predictions
```

| Column | Data Type | Description |
|---------|-----------|-------------|
| id | Integer | Primary key |
| timestamp | Timestamp | Prediction timestamp |
| amount | Double Precision | Transaction amount |
| transaction_type | VARCHAR(30) | Transaction category |
| fraud_probability | Double Precision | Predicted fraud probability |
| decision | VARCHAR(20) | Final business decision |

The table stores only production inference data and does not duplicate the original training dataset.

# 7. Prediction Logging Workflow

Every prediction generated by the API follows the workflow below.

```
Client Request
      │
      ▼
FastAPI
      │
      ▼
CatBoost Prediction
      │
      ▼
Decision Engine
      │
      ▼
log_prediction()
      │
      ▼
INSERT INTO predictions
      │
      ▼
PostgreSQL
```

Each prediction becomes part of the production history immediately after inference.

---

# 8. Database Operations

## Insert Operations

Following model inference, the API inserts the following information into PostgreSQL:

- Timestamp
- Transaction amount
- Transaction type
- Fraud probability
- Decision

Each prediction is committed as an independent database transaction.

---

## Read Operations

The database is queried by multiple platform components.

### Streamlit Dashboard

The monitoring dashboard retrieves stored predictions to display:

- Total predictions
- Approved transactions
- Verified transactions
- Blocked transactions
- Average fraud probability
- Recent prediction history

---

### Drift Detection

The drift detection module compares:

- Historical training data
- Production prediction data stored in PostgreSQL

This enables statistical comparison between training and production distributions.

# 10. Engineering Decisions

Several engineering decisions influenced the database architecture.

### Decision 1

A relational database was selected instead of CSV-based logging to provide persistent, queryable storage.

---

### Decision 2

Prediction logging was separated from the prediction logic.

This keeps inference responsibilities independent from persistence responsibilities.

---

### Decision 3

Database connections are created only when required and closed immediately after each operation.

This prevents unnecessary resource consumption.

---

### Decision 4

The database stores only production inference records rather than duplicating the complete training dataset.

This reduces storage requirements while preserving operational history.

# 11. Challenges

The primary challenges encountered during implementation included:

- Configuring PostgreSQL connectivity.

# 12. Lessons Learned

Integrating PostgreSQL demonstrated several important software engineering principles.

- Production systems require persistent storage.
- Machine learning services should preserve inference history.
- Relational databases simplify downstream analytics.
- A shared persistence layer improves modular system design.
