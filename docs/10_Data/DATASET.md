# 1. Purpose

This document describes the dataset used to develop the Adaptive Fraud Intelligence Platform. It provides an overview of the dataset, its features, data quality, preprocessing considerations, and challenges encountered before model development.

---

# 2. Background

Fraud detection systems rely on historical transaction records to learn patterns that distinguish legitimate transactions from fraudulent ones. The quality and characteristics of the dataset directly influence the performance and generalization capability of the machine learning model.

The dataset used in this project represents financial transaction records and is widely used for benchmarking fraud detection models.

---

# 3. Dataset Description

| Attribute | Value |
|-----------|-------|
| Dataset Name | Fraud Detection Dataset |
| Source | Kaggle |
| Domain | Financial Transactions |
| Total Records | ~6.36 Million |
| Total Features | 11 |
| Target Variable | isFraud |
| Learning Type | Supervised Binary Classification |

The dataset contains transaction-level information including transaction type, transaction amount, sender balance, receiver balance, and fraud labels.

| Feature | Description |

# 5. Dataset Characteristics

## Class Distribution

| Class | Count |
|------:|------:|
| Non-Fraud | 6,354,407 |
| Fraud | 8,213 |

The dataset is highly imbalanced, with fraudulent transactions representing only a very small fraction of the total records. This reflects a realistic fraud detection scenario encountered in financial systems.

---

## Missing Values

The dataset contains almost no missing values. During inspection, a negligible number of null entries (one record in several columns) was identified and handled during preprocessing.

---

## Duplicate Records

A duplicate check confirmed that no fully duplicated rows were present in the dataset.

---

# 6. Feature Selection

Not all dataset features were used during model training.

The following columns were excluded:

| Feature | Reason |
|----------|--------|
| nameOrig | Account identifier. Excluded because identifiers do not generalize to unseen customers and may lead to model overfitting. |
| nameDest | Destination account identifier. Excluded for the same reason as `nameOrig`; behavioral features are preferred over identifiers in production systems. |
| isFlaggedFraud | Rule-based indicator provided by the dataset. Excluded to prevent information leakage and allow the model to learn fraud patterns independently. |

---

# 7. Data Encoding

The categorical transaction type was converted into numerical values before model training using manual mapping.

| Transaction Type | Encoded Value |
|------------------|--------------:|
| PAYMENT | 0 |
| TRANSFER | 1 |
| CASH_OUT | 2 |
| DEBIT | 3 |
| CASH_IN | 4 |

This encoding ensured compatibility with the machine learning pipeline while preserving the distinction between transaction categories.

---

# 8. Class Imbalance Handling

The dataset exhibits severe class imbalance.

Instead of performing oversampling or undersampling, the CatBoost classifier was configured using:

```python
auto_class_weights="Balanced"
```

This allows CatBoost to automatically assign higher importance to minority class samples during training while preserving the original data distribution.

---

# 9. Challenges

Several data-related challenges were identified during dataset preparation:

- Extreme class imbalance.
- Presence of identifier columns that are unsuitable for production model training.
- Requirement for categorical feature encoding.
- Large dataset size requiring efficient preprocessing.

---

# 10. Lessons Learned

The dataset demonstrated that effective fraud detection depends not only on model selection but also on thoughtful data preparation.

Important lessons include:

- Real-world fraud datasets are naturally imbalanced.
- Identifier columns should be evaluated carefully before inclusion.
- Feature engineering has a significant impact on model performance.
- Data understanding is the foundation of the machine learning pipeline.

# 11. Future Improvements

These features are expected to improve model robustness while maintaining generalization capability.
